## 0. 현행 코드 진단 요약 (계획 수립의 전제)

현재 저장소에는 이미 동작하는 프로토타입이 있다. **버리고 새로 만들 필요는 없다.** 특히 `canvas-engine.js`의 의사(pseudo) 3D 투영 엔진(Z-buffer 정렬 + 원근 투영)은 PRD가 요구하는 수준을 충분히 만족하며, Three.js 도입 없이도 목표를 달성할 수 있다. 따라서 본 계획은 **기존 3파일 구조를 유지한 점진적 리팩터링**을 전제로 한다.

다만 아래 항목은 PRD/PDF와 명확히 어긋나 있어 계획의 출발점이 된다.

### 0.1 즉시 수정이 필요한 치명적 결함

| 구분 | 위치 | 내용 |
| :--- | :--- | :--- |
| **과학적 오류** | [atoms.js:91](../atoms.js#L91) | 산소(O)의 `stableElectrons: 8`. 중성 산소 원자의 전자 수가 이미 8개이므로, **아무 조작도 하지 않은 산소 원자가 즉시 "O²⁻ 성공" 판정**을 받는다. 정답은 10이어야 한다. |
| **과학적 오류** | [atoms.js:79](../atoms.js#L79) | 질소(N)의 `stableElectrons: 8`. 전자 1개만 얻어도 성공 처리되어 실제로는 N⁻를 N³⁻로 오판한다. 정답은 10. |
| **과학적 오류** | [atoms.js:103](../atoms.js#L103) | 플루오린(F)의 `stableElectrons: 8`. 전자를 **잃어야** 성공하는 상태가 되어 학습 목표와 정반대다. 정답은 10. |
| **규칙 위반** | [canvas-engine.js:224-243](../canvas-engine.js#L224-L243) | `findElectronAtPos()`가 모든 껍질을 순회한다. PRD 7.1의 "최외각 전자만 드래그 가능"이 전혀 강제되지 않아 **안쪽 K껍질 전자도 자유롭게 뽑을 수 있다.** |
| **모바일 미작동** | [canvas-engine.js:73](../canvas-engine.js#L73) | `touchend` 핸들러가 좌표 없이 `handleEnd()`를 호출한다. `clientX`가 `undefined`가 되어 **터치 환경에서는 전자를 버리는 동작이 아예 성립하지 않는다.** DoD 2번 항목의 정면 위반. |
| **모바일 미작동** | [canvas-engine.js:65-69](../canvas-engine.js#L65-L69) | `touchmove`를 `{ passive: false }`로 등록해 놓고 `preventDefault()`를 호출하지 않는다. 전자를 끄는 동안 페이지가 함께 스크롤된다. |
| **과학적 오류** | [atoms.js:319-323](../atoms.js#L319-L323) | 화합물 레시피에 **물(H₂O)이 이온 결합 화합물로 등재**되어 있다. 물은 공유 결합이며, PRD 11.3 목록에도 없다. |

### 0.2 PRD와의 데이터 불일치

- **B, C, Si**: PRD 4.3은 "공유 결합 안내 후 이온 만들기에서 제외"로 규정하나, 현행 코드는 B³⁺·C⁴⁺·Si⁴⁺라는 실존하지 않는 교육용 이온을 정답으로 제시한다.
- **He, Ne, Ar**: PRD 4.3은 "조작 비활성화"를 요구하나, 현행 코드는 자유 조작을 허용하고 "그냥 확인 버튼을 누르세요"라고 안내한다.
- **화합물 목록**: PRD 11.3의 8종 중 **K₂O와 Na₂O가 누락**되어 있고, 대신 PRD에 없는 CaO·Na₂S·K₂S·KCl·NaF·Li₂O·H₂O 7종이 추가되어 있다.
- **최소 정수비 판정 부재**: `checkCompoundCombination()`은 레시피와의 정확 일치만 검사한다. Na⁺ 2개 + Cl⁻ 2개를 넣으면 총전하는 0이지만 레시피에 없으므로 **아무 반응 없이 침묵**한다. PRD FR-11이 요구하는 "약분 유도 피드백"이 없다.

## 2. 구현 불가 / 범위 제외 기능

| 기능 | 판정 | 사유 |
| :--- | :--- | :--- |
| **교사용 학습 대시보드, 계정, 학급 관리** | ❌ 불가 | 서버·DB·인증이 필요하다. 현재는 백엔드가 없고, PRD 15는 MVP에서 로그인·학적 저장을 명시적으로 배제한다. 로컬 스토리지로는 여러 학생의 데이터를 교사가 볼 수 없다. |
| **랭킹 / 리더보드 (기기 간 공유)** | ❌ 불가 | 위와 동일. 기기 내 최고 점수 기록까지가 한계. |
| **오비탈 확률 구름 표현** | ❌ 제외 | PRD 17이 MVP 제외로 명시. 또한 PRD 4.4가 이 앱을 "행성 궤도 추상화 학습 모형"으로 규정했으므로, 오비탈 도입은 제품 콘셉트 자체와 충돌한다. |
| **공유 결합 시뮬레이션 (B, C, Si)** | ❌ 제외 | PRD 4.3은 해당 원소에 대해 **안내 문구 출력만** 요구한다. 시뮬레이션 구현이 아니라 **제외 처리가 정답**이다. 현행 코드의 B³⁺/C⁴⁺/Si⁴⁺는 제거 대상. |
| **다원자 이온 (CO₃²⁻, SO₄²⁻, OH⁻, NH₄⁺)** | ❌ 제외 | PDF p.13이 Version 3로 지정. 현행 단원자 껍질 모형으로는 표현 구조가 성립하지 않아 데이터 모델 전면 재설계가 필요하다. |
| **결정 격자 시각화 / 전기 전도성 시뮬레이션** | ❌ 제외 | Version 3. 현행 단일 원자 캔버스 엔진으로는 다중 입자 격자 렌더링이 불가능하며, 성능 목표(보급형 태블릿)와도 충돌한다. |
| **전이금속 이온 (Sc~Zn 등) 및 21번 이상 원소** | ❌ 제외 | PDF p.5의 주기율표 목업에 Sc·Ti·V·Rb·Cs 등이 그려져 있으나 이는 **UI 배치를 보여주기 위한 플레이스홀더**다. PRD 4.1이 대상을 1~20번으로 한정하므로 실제 주기율표는 1~20번만 구성한다. 전이금속은 단순 옥텟 모형으로 설명 불가. |
| **수용액 이온화 / 수화 현상** | ❌ 제외 | Version 3. |
| **AI 챗봇 튜터** | ❌ 제외 | PRD 17 MVP 제외 명시. |
| **멀티플레이 / 협동 실험** | ❌ 제외 | PRD 17 MVP 제외 명시. 실시간 서버 필요. |
| **실제 3D 엔진 수준의 물리 시뮬레이션** | ⚠️ 대체 | Three.js/WebGL 도입은 가능하나 **불필요하다.** 현행 Canvas 2D 투영 엔진이 PDF p.5의 요구(원자핵 중심 고정, 최대 4개 껍질, 자유 회전·확대)를 모두 충족하고, 보급형 태블릿 성능 목표에는 오히려 유리하다. **의사 3D 유지를 권고한다.** |
| **효과음 에셋 기반 사운드** | ⚠️ 대체 | 현행 Web Audio API 신시사이저로 충분. 별도 오디오 에셋 제작은 범위 밖. On/Off 토글만 추가한다. |
| **PDF p.5의 '중성자 수' 정확 표기** | ⚠️ 조건부 | 구현 가능하나, 중성자 수는 동위원소마다 다르다. **가장 흔한 동위원소 기준 고정값**으로 데이터에 넣고, 원자핵 렌더링은 PRD 10.1에 따라 단순 구(Sphere)로 유지한다. |

## 3. 개발 계획

### 3.1 원칙

1. **데이터 → 규칙 → UI → 피드백** 순으로 쌓는다. 과학적으로 틀린 데이터 위에 UI를 얹으면 전부 다시 짜야 한다.
2. 기존 3파일 구조(`atoms.js` 데이터 / `canvas-engine.js` 렌더·입력 / `app.js` 상태·UI)를 유지하되, **판정 로직을 `rules.js`로 분리**한다. 현재 판정이 `app.js`의 `checkOctetRule()` 한 함수에 박혀 있어 모드 분기와 단위 테스트가 불가능하다.
3. 각 단계는 **PRD 19장 수용 기준의 해당 항목이 통과**해야 완료로 본다.

### 3.2 단계별 계획

#### Phase 0 — 데이터 정합성 확보 (블로커)

> **목표**: PRD 4장과 100% 일치하는 원소·화합물 데이터 확보. 이후 모든 작업의 전제.

- `atoms.js` 스키마를 PRD 13.1 / PDF p.12 규격으로 마이그레이션
  - `shells` → `neutralShells`, `stableElectrons` → `targetShells` + `targetElectronChange` + `targetCharge`
  - `isIonActivityEnabled`, `ruleType`(octet/duet), `ionType`(cation/anion), `disabledReason`, `neutronCount` 추가
- **N·O·F 정답 전자 수 교정 (8 → 10)** ← 최우선
- B·C·Si → `isIonActivityEnabled: false` + 공유 결합 안내 문구
- He·Ne·Ar → `isIonActivityEnabled: false` + 비활성 기체 안내 문구
- H → 듀엣 규칙(`ruleType: "duet"`)으로 분리, H⁺만 기본 제공
- 화합물 레시피를 PRD 11.3의 8종으로 재구성 (H₂O 제거, K₂O·Na₂O 추가)
- **완료 기준**: 20개 원소 전부에 대해 `양성자 − 목표 전자 = 목표 전하` 항등식이 성립. 산소를 선택하고 아무것도 하지 않았을 때 성공 판정이 나지 않는다.

#### Phase 1 — 조작 규칙 엔진

> **목표**: PRD 7장의 조작 규칙을 코드로 강제. "할 수 없는 것"을 확실히 막는다.

- `rules.js` 신설: `getValenceShellIndex()`, `canDragElectron()`, `canDropElectron()`, `evaluateState()`, `diagnoseError()`
- `canvas-engine.js`의 히트테스트를 최외각 껍질로 제한, 내각 전자는 시각적으로도 잠금 표현(채도 저하 + 자물쇠 테두리)
- 껍질 수용량 초과 드롭 차단
- 전자 보관함 상태를 실제 배열로 관리 (제거 → 보관함 적립, 추가 → 보관함에서 차감)
- **터치 입력 정상화**: `changedTouches` 좌표 전달, `touch-action: none`, `preventDefault()` 적용
- **완료 기준**: PRD 19 — "Ne 등 비활성 기체는 조작이 아예 비활성화된다", 내각 전자 드래그 불가. 태블릿 실기기에서 전자 제거·추가가 마우스와 동일하게 동작.

#### Phase 2 — 화면 레이아웃 재구성

> **목표**: PDF p.5의 4분할 인터페이스 구현.

- 좌측: 1~20번 미니 주기율표 (실제 주기율표 배치 형태, 금속/비금속/비활성 기체 범례, 학습 가능/설명 전용 시각 구분)
- 중앙: 3D 캔버스 + 확대/축소 컨트롤 + '원래 위치로' + 회전 On/Off
- 우측: 실시간 정보 패널 (기호·이름·원자번호 / 양성자·중성자·전자 수 / 현재 전하 / K·L·M·N 껍질별 전자 배치)
- 하단: 전자 보관함
- 상단 공통: 탭 · 도움말 · 소리 · 처음부터
- 모형의 한계 안내 상시 배치
- **완료 기준**: PDF p.5의 ①②③④ 영역이 모두 존재하고 실시간 동기화된다.

#### Phase 3 — 판정·피드백·힌트

> **목표**: "틀렸습니다"가 아니라 "왜 틀렸는지 스스로 깨닫게" 만든다. (PDF p.7)

- 오류 유형 6종 진단 분기 (PRD 8.2)
- 3단계 점진 힌트 (PRD 8.3) — 요청할 때마다 단계 상승, 원소별 3단계 문구는 데이터에서 생성
- 성공 판정 4조건 전부 검사 (전자 수 / 전자 배치 / 이동 방향 / 내각 조작 없음)
- 결과 패널에 전하 계산식 명시 (`양성자 11 − 전자 10 = +1`) — PDF p.8의 저울 비유 반영
- **완료 기준**: PRD 19 — Na에 전자 주입 시 방향 오류 피드백, Mg 1개 제거 시 미완성 판정, Cl 제거 시도 시 오답 가이드.

#### Phase 4 — 화합물 만들기 탭

> **목표**: PDF p.10의 '총전하 0 + 최소 정수비' 2중 검증.

- 이온 만들기 / 화합물 만들기 탭 분리
- 양이온·음이온 카드 목록 → 중앙 결합 영역 드래그
- 총전하 실시간 계산 (이미 있음, 유지)
- **최소 정수비 검증**: 총전하 0 달성 후 이온 개수의 최대공약수가 1인지 검사
- 오답 피드백 4종 (PRD 11.5)
- 화학식 렌더링 규칙 (계수 1 생략, 아래첨자)
- **완료 기준**: PRD 19 — Na⁺1 + Cl⁻1은 NaCl 완성, Na⁺2 + Cl⁻2는 최소 정수비 위반으로 차단되고 **약분 유도 메시지가 출력된다.**

#### Phase 5 — 학습 모드 · 접근성 · 반응형

- 자유 탐색 모드 (판정 없음, 실시간 전하만 관찰) / 연습 모드 (목표 제시 + 힌트·진단 활성)
- 접근성: 양성자(빨강·구·`+`) / 중성자(회색·구·무표기) / 전자(파랑·작은 구·`−` + 최외각 강조 테두리) — 색상 외 최소 2단서
- 3D 회전 On/Off, 폰트 최소 크기, 위첨자/아래첨자 렌더링 검증
- 반응형: 데스크톱 3분할 → 태블릿 2분할 → 모바일 세로 스택
- CDN 의존성 제거 (오프라인 학교망 대비)
- **완료 기준**: PRD 15 비기능 요구사항 전항 충족.

#### Phase 6 — 검증

- 20개 원소 × (자유/연습) 전수 시나리오 점검
- 화합물 8종 정답 + 오답 케이스(전하 불균형·약분 미필) 점검
- 보급형 태블릿 Chrome 실기기 성능 측정 (목표: 30fps 이상, 초기 로딩 3초 이내)
- PRD 20장 완료 정의 5개 항목 체크
- **완료 기준**: PRD 19·20 전항 통과.

### 3.3 의존 관계

```
Phase 0 (데이터)  ──┬─→ Phase 1 (규칙) ──→ Phase 3 (판정/힌트) ──┐
                    │                                            ├─→ Phase 6 (검증)
                    ├─→ Phase 2 (레이아웃) ──→ Phase 5 (모드/접근성) ─┤
                    │                                            │
                    └─→ Phase 4 (화합물) ────────────────────────┘
```

Phase 0은 모든 작업의 블로커다. Phase 2와 Phase 4는 Phase 1 완료 후 병행 가능하다.

| 항목 | 결정 | 근거 |

## 8. 구현 중 조정한 결정

계획 수립 시점에는 보이지 않았던 문제 두 가지를 구현 중에 만나 아래와 같이 조정했다.

### 8.1 화합물 이름표를 15종으로 확장 (PRD 11.3의 8종은 그대로 유지)

PRD 11.3은 MVP 화합물을 8종으로 지정한다. 그런데 활동에 필요한 이온 카드(양이온 5종 ×
음이온 3종)로는 **15가지 조합**이 만들어지고, 그중 7가지(NaF, KCl, KF, CaO, MgF₂, AlCl₃,
AlF₃)는 총전하 0과 최소 정수비를 모두 만족하는 **화학적으로 올바른 이온 화합물**이다.

8종만 인정하면 학생이 정답을 만들어도 "목록에 없다"며 침묵하게 된다. 옳게 한 조작을
오답처럼 취급하는 것은 PRD 8.1(오답 원인을 스스로 깨닫게 한다)의 정신에 어긋난다.
따라서 15종 전부에 이름표를 붙이되, PRD의 8종에만 `isMvpTarget: true`를 두어
**체크리스트와 진도 표시는 PRD 8종만** 대상으로 한다. 나머지 7종은 만들면 축하하되
진도에는 포함하지 않는다.

### 8.2 미니 주기율표 셀에서 한글명 제거

Blueprint p.5의 좌측 패널 폭(272px)에 8열을 넣으면 셀 하나가 30px 남짓이다. 여기에
한글명을 넣으면 7.5px가 되어 읽히지 않는다. Blueprint p.3의 주기율표 역시 번호와 기호만
싣고 있다. 따라서 셀에는 **번호 + 기호**만 두고, 한글명은 hover 툴팁 · `aria-label` ·
선택 시 우측 정보 패널에서 제공한다.

### 8.3 덤으로 추가한 것 — 해시 라우팅

`#compound` / `#ion` 해시로 특정 활동에 바로 진입할 수 있다. 헤드리스 브라우저에서
화합물 탭을 검증하려다 넣었으나, 교사가 수업 자료에 특정 활동을 직접 링크할 수 있어
그대로 남겼다.

## 9. 남은 작업

MVP 범위에서 의도적으로 제외했거나, 실기기 확인이 필요한 항목이다.

- **보급형 태블릿 실기기 성능 측정** — Phase 6의 성능 항목은 헤드리스 환경에서 측정할 수
  없다. 실제 학교 태블릿에서 30fps·초기 로딩 3초를 확인해야 한다. 저사양 감지 시
  그림자·파티클을 끄는 옵션은 아직 넣지 않았다(리스크 표 4행).
- **실기기 터치 확인** — 통합 테스트는 합성 `TouchEvent`로 검증했다. 실제 손가락 입력의
  히트 정밀도(터치 반경 32px 설정)는 기기에서 봐야 한다.
- **v2 이관 항목** — 도전 모드, 타임 어택, 스코어링, 학습 리포트, 오답 노트, 교사용 기능.
